<h2>1) Definitions &amp; examples</h2>

<h3>Iterative Development</h3>
<ul>
  <li><strong>Concept:</strong> Work on the <em>whole system</em> in rough form, then <strong>repeat</strong> to increase <strong>detail &amp; quality</strong>.</li>
  <li><strong>Analogy:</strong> A <em>sculptor</em> shaping the same statue through passes.</li>

<h3>Incremental Development</h3>
<ul>
  <li><strong>Concept:</strong> Build <strong>usable, done</strong> slices (increments) piece by piece.</li>
  <li><strong>Analogy:</strong> Finish one <strong>room</strong> before adding another; each is usable in its scope.</li>
</ul>

<h2>2) Scrum synergy: Iterative + Incremental</h2>
<ul>
  <li><strong>Incremental</strong>: each <strong>Sprint</strong> delivers a <strong>DoD-complete</strong> Increment.</li>
  <li><strong>Iterative</strong>: the team also <strong>refines</strong> existing capabilities based on feedback.</li>
  <li><strong>Rule of thumb</strong>: cut <strong>vertical slices</strong> across UIâ€“APIâ€“DB, not horizontal layers only.</li>

<h3>â€œCookbookâ€ for thin vertical slices</h3>

  <li>Start with the <strong>happy path</strong>, add edge cases later.</li>
  <li>Slice by <strong>workflow step</strong>, <strong>persona</strong>, or <strong>channel</strong> only when truly helpful.</li>
  <li>Favor slices with <strong>measurable impact</strong> (events/analytics).</li>
</ul>

<h2>3) When to favor Iterative vs Incremental?</h2>

<h3>Favor Iterative whenâ€¦</h3>
<ul>
  <li>Architecture/UX is <strong>unclear</strong>; you must shape the whole first.</li>

  <li><strong>R&amp;D/Prototype</strong> requires fast whole-system trials.</li>

<h3>Favor Incremental whenâ€¦</h3>
<ul>
  <li>You need <strong>early time-to-value</strong> (MVP).</li>
  <li>Features/modules can be <strong>released independently</strong>.</li>
  <li>You must <strong>measure</strong> feature-level impact.</li>
