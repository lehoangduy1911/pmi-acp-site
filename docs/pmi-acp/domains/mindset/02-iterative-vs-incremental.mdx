<h2>Summary (EN)</h2>
<p><strong>Iterative</strong> = <em>refine the whole</em> (coarse â†’ detailed over cycles). <strong>Incremental</strong> = <em>add usable parts</em> (each piece is complete and valuable). <strong>Agile</strong> blends both: each <em>Sprint</em> ships a <strong>usable increment</strong> while <strong>iteratively</strong> refining what already exists via feedback.</p>

<h2>1) Quick contrast</h2>

<h3>Iterative Development</h3>
<ul>
  <li><strong>Concept:</strong> Work on the <em>entire product</em> in rough form, then <strong>repeat to refine</strong> until done.</li>
  <li><strong>Analogy:</strong> A <em>sculptor</em> shaping the same statue through multiple passes.</li>
  <li><strong>Example:</strong> Car design: iteration 1 rough clay model; iteration 2 refine body lines; iteration 3 add interior detailsâ€”<em>the whole car</em> gets <strong>better each pass</strong>.</li>
  <li><strong>Key:</strong> Focus on <em>refinement</em>, increasing <strong>detail &amp; quality</strong> each iteration.</li>
</ul>

<h3>Incremental Development</h3>
<ul>
  <li><strong>Concept:</strong> Build the product <strong>piece by piece</strong>. Each <strong>increment</strong> is <strong>complete &amp; usable</strong>.</li>
  <li><strong>Analogy:</strong> <em>House-building</em> by finished sections: foundation â†’ one roomâ€™s walls â†’ that roomâ€™s roofâ€¦</li>
  <li><strong>Example:</strong> Car assembly by modules: increment 1 = running chassis + engine; increment 2 = complete body shell; increment 3 = finished interiorâ€”<em>each increment is done</em> in its scope.</li>
  <li><strong>Key:</strong> Emphasis on <em>delivery</em> of <strong>usable value slices</strong>.</li>
</ul>

<h2>2) The synergy: Iterative + Incremental in Agile</h2>
<p><strong>Agile</strong> combines both so you <em>ship value early</em> (incremental) and <em>improve quality continuously</em> (iterative).</p>
<ul>
  <li><strong>Sprint 1 (Increment 1):</strong> Login + balance (usable).</li>
  <li><strong>Sprint 2 (Increment 2):</strong> Add money transfer (new value).</li>
  <li><strong>Iterative refinement:</strong> Improve the login UI and performance based on feedback.</li>
</ul>

<h2>2.1) When to favor Iterative vs Incremental?</h2>

<h3>Prefer Iterative whenâ€¦</h3>
<ul>
  <li>The overall design/architecture is fuzzy; you must <strong>shape the whole</strong> first.</li>
  <li>You need to <strong>improve quality/detail</strong> over multiple feedback cycles.</li>
  <li>R&amp;D/UX work requires <strong>rapid trial-and-refinement</strong> at a system level.</li>
</ul>

<h3>Prefer Incremental whenâ€¦</h3>
<ul>
  <li>You aim for <strong>early time-to-value</strong>: ship <strong>value slices</strong> that users can use now.</li>
  <li>The system allows <strong>feature/module independence</strong> to release in parts.</li>
  <li>You want to <strong>measure impact</strong> feature by feature (A/B, events, funnels).</li>
</ul>

<h3>Blend in Scrum</h3>
<ul>
  <li>Each <strong>Sprint</strong> ships a <strong>usable increment</strong> (Incremental) while <strong>refining</strong> existing work (Iterative).</li>
</ul>

<h2>3) Comparison vs traditional &amp; adoption barriers</h2>

<h3>Comparison table</h3>

| Characteristic | Traditional (Waterfall) | Incremental | Iterative | Agile (Iterative & Incremental) |

| Delivery | One big at the end | Incremental deliveries | One big, refined over time | Frequent, small, usable deliveries |
| Feedback | End of project | Per increment | Per iteration | Continuous throughout |
